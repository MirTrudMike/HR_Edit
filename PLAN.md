# План разработки инструмента для извлечения текста из DOCX

## Цель

Сделать инструмент, который принимает Word-документ `.docx`, извлекает из него полный текст в правильном порядке и распознает текст внутри встроенных изображений, включая формулы. Результат должен быть пригоден для дальнейшего редактирования: сначала достаточно Markdown или TXT с понятной разметкой блоков.

## Baseline

Это первая зафиксированная рабочая версия прототипа.

Что уже работает:

- `docx_extract.py` обрабатывает один `.docx` и извлекает текст, таблицы, inline-изображения и OCR;
- `archive_extract.py` обрабатывает ZIP-архивы с DOCX внутри;
- `work/runs/`, `work/archive_runs/` и `work/cache/ocr/` хранят временные результаты и кэш;
- `index.html` для архива - одна страница с деревом файлов, preview и копированием `Markdown`/`Text`;
- preview поддерживает сворачивание папок и локальный zoom;
- `--describe-diagrams` описывает крупные нетекстовые изображения;
- по умолчанию debug JSON и извлеченные картинки не засоряют папку результата архива.

Главные команды:

```bash
python3 docx_extract.py test.docx --ocr hybrid --format md
python3 archive_extract.py "/path/to/archive.zip" --format md --ocr hybrid --describe-diagrams --clean
```

## Исходный сценарий

Документы скачиваются из рабочего кабинета компании как вложенные Word-файлы. На странице есть прямые ссылки на `.docx`. В будущем возможен локальный Chrome extension в режиме разработчика: он будет находить ссылки на документы на странице, скачивать выбранный файл и запускать распознавание.

На первом этапе расширение не делаем. Сначала строим и проверяем надежный алгоритм на локальном Python-инструменте.

## Наблюдения по test.docx

- `.docx` является zip-контейнером с XML-разметкой Word.
- Основной текст лежит в `word/document.xml`.
- Встроенные изображения связаны через `word/_rels/document.xml.rels`.
- Порядок текста и картинок можно восстановить точно: изображения находятся внутри `w:r` в нужном месте абзаца.
- Не все изображения имеют корректное расширение: в примере `media/image.bin` фактически является PNG.
- В примере формулы `30°` и `30° · 3 = 90°` вставлены как маленькие PNG-картинки прямо внутри строки объяснения.

## Базовый алгоритм

1. Открыть `.docx` как zip-архив.
2. Прочитать `word/document.xml`.
3. Прочитать связи из `word/_rels/document.xml.rels`.
4. Парсить XML через `lxml`.
5. Рекурсивно обходить документ:
   - абзацы;
   - таблицы;
   - строки таблиц;
   - ячейки;
   - вложенные таблицы;
   - runs внутри абзацев.
6. Для обычного текста брать содержимое `w:t`.
7. Для переносов строк учитывать `w:br`.
8. Для изображений:
   - найти `r:embed`;
   - по relationship ID найти файл изображения;
   - прочитать байты изображения;
   - определить реальный тип через Pillow;
   - передать изображение в OCR;
   - вставить распознанный текст на место изображения.
9. Сформировать Markdown/TXT с сохранением структуры документа.
10. Сохранить диагностический JSON: какие изображения найдены, где они стояли, что распознано, confidence/статус.

## OCR-стратегия

### Локальный слой

Использовать локальные библиотеки для простых и дешевых случаев:

- `Pillow` для чтения и нормализации изображений.
- `lxml` для XML.
- Возможно `PaddleOCR` или `Tesseract` для обычного текста на картинках.

Минус: формулы и мелкие математические PNG могут распознаваться нестабильно.

### OpenAI API слой

Использовать OpenAI API для сложных изображений:

- маленькие формулы;
- смешанный текст и математические символы;
- изображения, где локальный OCR дал низкую уверенность;
- случаи, где нужно вернуть не только текст, но и нормализованную формулу.

По текущей документации OpenAI, актуальные модели поддерживают text input, image input, text output и vision через Responses API. Это подходит для задачи OCR/извлечения текста из картинок.

Рекомендуемый режим для MVP:

- сначала извлекать изображения из DOCX локально;
- отправлять в OpenAI только сами картинки, а не скриншот всей страницы;
- просить модель вернуть строго JSON:
  - `kind`: `text`, `formula`, `diagram`, `empty`, `unknown`;
  - `text`: распознанный текст;
  - `latex`: LaTeX, если это формула;
  - `confidence`: оценка уверенности;
  - `notes`: короткое пояснение при сомнении.

Для вставки в итоговый Markdown:

- если `kind = formula`, вставлять текстовую форму, например `30° · 3 = 90°`;
- опционально рядом хранить LaTeX в диагностике;
- если `kind = diagram`, оставлять маркер вроде `[Изображение: циферблат]`, не превращая схему в шумный OCR.

## Формат результата

Первый формат результата: Markdown.

Пример:

```markdown
## Текст задания

(Изображение 1)

Минутная стрелка перешла с отметки 12 на указанную цифру. Какой угол она описала?

Заполни таблицу, выбрав правильный ответ.

| С 12 на 3 | С 12 на 6 | С 12 на 9 | С 12 на 12 (полный круг) |
| --- | --- | --- | --- |
| <1>° | <2>° | <3>° | <4>° |

Изображение 1 - [IMAGE: media/image.bin]

## Объяснение решения

Циферблат делится на 12 частей по 30°.

Например, от 12 до 3 — три части: 30° · 3 = 90°
```

Дополнительно сохранять `*.ocr.json` для отладки.

## Этапы работы

### Этап 1. Прототип парсера DOCX

Цель: получить полный текст документа с маркерами изображений в правильных местах.

Задачи:

- разобрать `word/document.xml`;
- восстановить relationships;
- пройти по абзацам и таблицам;
- сохранить порядок текста, таблиц и картинок;
- вывести Markdown без OCR, например `[IMAGE: media/image2.png]`.

Критерий готовности: для `test.docx` изображение `30°` стоит между `по` и точкой, а `30° · 3 = 90°` стоит после фразы `три части:`.

### Этап 2. Извлечение и нормализация изображений

Цель: подготовить картинки к распознаванию.

Задачи:

- извлекать все embedded images;
- определять формат по содержимому;
- сохранять копии изображений в рабочую папку;
- увеличивать маленькие изображения перед OCR;
- добавлять белый фон для прозрачных PNG;
- сохранять размеры и исходный relationship ID.

Критерий готовности: `image.bin`, `image2.png`, `image3.png` корректно открываются и имеют понятные метаданные.

### Этап 3. OCR через OpenAI API

Цель: распознать текст/формулы на изображениях и вставить их в Markdown.

Задачи:

- сделать модуль `ocr_openai.py`;
- передавать изображение в модель с image input;
- требовать структурированный JSON;
- добавить retry и логирование ошибок;
- кэшировать результат OCR по hash изображения, чтобы не платить повторно за одинаковые картинки;
- настроить `.env` с `OPENAI_API_KEY`.

Критерий готовности: для `test.docx` получаем `30°` и `30° · 3 = 90°` в нужных местах.

### Этап 4. Классификация изображений

Цель: не пытаться распознавать как текст все подряд.

Задачи:

- различать формулы, обычный текст, схемы/рисунки и пустые картинки;
- для схем оставлять маркер изображения или краткое описание;
- добавить настройку режима:
  - `text-only`;
  - `text-plus-image-markers`;
  - `text-plus-image-descriptions`.

Критерий готовности: циферблат не превращается в случайный набор цифр, а остается как понятный маркер изображения.

### Этап 5. CLI-инструмент

Цель: получить удобный локальный запуск.

Команда:

```bash
python docx_extract.py test.docx --out test.md
```

Опции:

- `--format md|txt|json`;
- `--ocr none|openai|local|hybrid`;
- `--images-dir extracted_images`;
- `--debug-json test.ocr.json`;
- `--keep-image-markers`;
- `--describe-diagrams`.

Критерий готовности: можно обработать один файл без ручного вмешательства.

### Этап 6. Пакетная обработка

Цель: обрабатывать папку документов.

Команда:

```bash
python batch_extract.py ./docs --out ./output
```

Задачи:

- обработка всех `.docx` в папке;
- отчет об ошибках;
- повторное использование OCR-кэша;
- сохранение исходного имени файла в результате.

### Этап 7. Подготовка к Chrome extension

Цель: понять минимальную архитектуру будущего расширения.

Предпочтительный вариант:

- Chrome extension находит ссылки на `.docx` на странице.
- По клику отправляет URL в локальный Python-сервис.
- Локальный сервис скачивает файл, распознает его и возвращает Markdown/TXT.

Почему так:

- API-ключ OpenAI не хранится в расширении;
- Python-код и зависимости остаются локально;
- проще работать с DOCX, OCR-кэшем и файлами;
- расширение остается тонким интерфейсом.

Возможная локальная архитектура:

```text
Chrome extension
    -> http://127.0.0.1:8765/extract?url=...
        -> Python FastAPI service
            -> download DOCX
            -> parse DOCX
            -> OCR images
            -> return Markdown/TXT
```

### Этап 8. Интерфейс

Решить после стабильного прототипа.

Возможные варианты:

## Риски

- В Word могут быть не только inline images, но и floating images/text boxes. Их придется отдельно учитывать.
- Формулы могут быть не картинками, а Office Math (`m:oMath`). Их лучше извлекать напрямую из XML, без OCR.
- Некоторые документы могут быть `.doc`, а не `.docx`. Для них нужен отдельный конвертер через LibreOffice.
- OCR может ошибаться на мелких формулах; потребуется нормализация изображения и проверка confidence.
- Если документ содержит конфиденциальные данные, использование внешнего API должно быть согласовано.
